Each step answered a tension raised by the previous one. Tables were invented
/because/ shells needed somewhere to put shared state. Partisan was adopted
/because/ the network topology was unreliable. NetworkMonitor was introduced
/because/ Partisan alone couldn't survive WiFi hops. Antifragile storm handling
emerged /because/ peer churn made broadcast amplification catastrophic. The
sequence is irreversible in its logic: you cannot start at step ⑧.

[[file:diagrams/visual-join.png]]

The =Table= is a /singleton per Disciple per book/, partitioned across
schedulers by =PartitionSupervisor= keyed on the LiveView pid. It accepts
additional watchers (multiple browser tabs of the same user) without
duplicating state. Race condition handling is explicit: between the Registry
lookup and the start_child call, another tab may win — the =:already_started=
reply falls through to =add_watcher=.

** Dynamics — NetworkMonitor Hot-Swap

When the user's WiFi flaps or they roam to a new network, the BEAM does /not/
restart. A choreographed sequence re-homes the node to its new address without
dropping its identity.

#+BEGIN_SRC mermaid :file diagrams/visual-hotswap.png
sequenceDiagram
  autonumber
  participant NM as NetworkMonitor<br/>(3s poll)
  participant MDNS
  participant PC as :partisan_config
  participant HV as HyParView<br/>Manager
  participant PS as partisan_sup<br/>(acceptor pool)
  participant PT as :persistent_term

  loop every 3s
    NM->>NM: new_ips = routable_ipv4_addrs()
    alt IPs unchanged
      NM->>NM: reset debounce
    else IPs differ
      NM->>NM: debounce (need 2 stable reads)
    end
  end

  Note over NM: confirmed IP change
  NM->>MDNS: goodbye(old_ips)  [TTL=0 ×2]
  NM->>PC: set(:listen_addrs, new_addrs)
  NM->>HV: refresh_node_spec  [fixes stale JOIN identity]
  NM->>PS: terminate + restart acceptor_socket_pool_sup
  alt bind ok
    NM->>NM: disconnect_stale_peers (leave, not disconnect)
    NM->>MDNS: rejoin multicast on new ifaces
    NM->>MDNS: reannounce(new_ips)
    NM->>PT: put({Gate, :addr}, new_addr)
    NM->>MDNS: on_network_change (reset failure tracking)
  else bind fails
    NM->>PC: revert listen_addrs to old
    Note over NM: retry next poll
  end
#+END_SRC

#+RESULTS:
[[file:diagrams/visual-hotswap.png]]

Three pieces of non-obvious knowledge live in this sequence:

1. /Goodbye before config update/ — packets must leave the interface that
   peers still recognize, or they arrive on the new network and are dropped
   as unknown.
2. /Refresh HyParView node_spec/ — the manager caches =node_spec()= at init;
   without refresh, outgoing JOINs carry dead IPs and peers never wire up
   the reverse connection Phoenix.PubSub needs.
3. /=leave= not =disconnect=/ — leave purges active AND passive views;
   disconnect only kills TCP, leaving =random_promotion= to rediscover dead
   IPs every 3s forever.

** Attractor Map — Where State Lives

Some state is /accumulated/ (reservoirs, survive churn). Some state is
/flowing/ (conduits, exist only in transit). Others are /ephemeral/ (cached,
rebuildable). Knowing which is which tells you what a crash costs.

#+BEGIN_EXAMPLE
  ┌──────────── RESERVOIRS (accumulate, cost on loss) ──────────────┐
  │                                                                 │
  │   Dojo.Table  ─────►  per-Disciple tuplespace, watcher list     │
  │   Dojo.Gate   ─────►  presence metadata (who+where+addr)        │
  │   :persistent_term ─► addr cache, read-hot, never copied        │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
  ┌──────────── CONDUITS (flow, crash is safe) ─────────────────────┐
  │                                                                 │
  │   Phoenix.PubSub messages   (fan-out, no history)               │
  │   Partisan plumtree gossip  (eventual delivery)                 │
  │   mDNS multicast packets    (self-describing, rebroadcastable)  │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
  ┌──────────── EPHEMERAL (cached, rebuildable) ────────────────────┐
  │                                                                 │
  │   Dojo.Cache    ─► Nebulex partitioned over schedulers          │
  │   MDNS peer cache ─► rebuilt next poll cycle (5s)               │
  │   HyParView views ─► re-converge after JOIN                     │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
#+END_EXAMPLE

Design rule implied by the map: /reservoirs must have a restart strategy/,
/conduits must not own state/, /ephemeral stores must tolerate eviction/.
When these rules are respected, "let it crash" is safe.

*Leverage points* (Meadows, from weakest to strongest):
- /Parameters/ — poll intervals (3s, 5s, 100ms debounce) — tunable
- /Stocks and flows/ — Table TTL, announce backoff cap — structural
- /Information flows/ — Presence metadata as transport — paradigm
- /Rules/ — Plumtree handler contract 
- /System goals/ — for e.g. "survive WiFi roaming"
The highest leverage in Dojo's history came from changing /rules/ (fix
handler contract) and /goals/ (make it roam-survivable), not from tuning
parameters.

**** Cells as an idea
The original conception of it had the following parts.



  - I thought of objects being like biological cells and/or individual computers on a network, only able to communicate with messages (so messaging came at the very beginning -- it took a while to see how to  do messaging in a programming language efficiently enough to be useful).



  - I wanted to get rid of data. The B5000 almost did this via its  almost unbelievable HW architecture. I realized that the  cell/whole-computer metaphor would get rid of data, and that "<-"  would be just another message token (it took me quite a while to  think this out because I really thought of all these symbols as names for functions and procedures.



  - My math background made me realize that each object could have  several algebras associated with it, and there could be families of  these, and that these would be very very useful. The term  "polymorphism" was imposed much later (I think by Peter Wegner) and  it isn't quite valid, since it really comes from the nomenclature of  functions, and I wanted quite a bit more than functions. I made up a  term "genericity" for dealing with generic behaviors in a  quasi-algebraic form.

    
My original experiments with this architecture were done using a 

model I adapted from van Wijngaarten's and Wirth's "Generalization of 

Algol" and Wirth's Euler. Both of these were rather LISP-like but 

with a more conventional readable syntax. I didn't understand the 

monster LISP idea of tangible metalanguage then, but got kind of 

close with ideas about extensible languages draw from various 

sources, including Irons' IMP.



The second phase of this was to finally understand LISP and then 

using this understanding to make much nicer and smaller and more 

powerful and more late bound understructures. Dave Fisher's thesis 

was done in "McCarthy" style and his ideas about extensible control 

structures were very helpful. Another big influence at this time was 

Carl Hewitt's PLANNER (which has never gotten the recognition it 

deserves, given how well and how earlier it was able to anticipate 

Prolog).



The original Smalltalk at Xerox PARC came out of the above. The 

subsequent Smalltalk's are complained about in the end of the History 

chapter: they backslid towards Simula and did not replace the 

extension mechanisms with safer ones that were anywhere near as 

useful.
A BEAM process is not an object, not a thread, not a service. It is closer to
a /cell/: an enclosed unit of state with a semi-permeable membrane, a nucleus
that transforms incoming signals into outgoing ones, and a lifecycle governed
by an external organ (its supervisor). When many cells coordinate by signals
alone — no shared memory, no locks, no ambient authority — you get tissue.
Tissues make organs. Organs make a /system/.

This section dissects Dojo's key cells: what their membrane looks like, what
lives in their nucleus, what signals they accept, and how they die.

Design rule that follows: /what a cell exposes is not what it is/. The membrane
is the design surface; the nucleus is private. You can replace the nucleus
entirely (rewrite the GenServer's state) as long as the signals remain the
same shape. This is why Alexander's "positive space between modules" applies
here literally — the contract IS a designed object.

** =Dojo.Table= — Singleton Tuplespace Cell
:PROPERTIES:
:MODULE: Dojo.Table
:END:

#+transclude: [[file:lib/dojo/table.ex]]  :lines 1-30 :src elixir

One cell per Disciple per book. A memory of what that learner has drawn,
said, or computed, with a 10-minute forgetfulness if no tab is watching.

#+BEGIN_EXAMPLE
  ┌───────────────────────────── Dojo.Table ────────────────────────────┐
  │                                                                     │
  │  MEMBRANE                                                           │
  │  ─────────                                                          │
  │   publish(pid, msg, event)       → cast  {:publish, msg, event}     │
  │   add_watcher(pid, lv_pid)       → call  {:add_watcher, lv_pid}     │
  │   change_meta(pid, {k, v})       → call  {:change_meta, delta}      │
  │   last({topic, node}, event)     → call  {:last, event} [local/RPC] │
  │                                                                     │
  │  NUCLEUS                                                            │
  │  ───────                                                            │
  │   %{ watchers :: MapSet<lv_pid>,                                    │
  │      topic, reg_key, disciple,                                      │
  │      last :: %{event ⇒ store},   ← in-process latest                │
  │      broadcast_timer, pending_broadcast }                           │
  │                                                                     │
  │  TRANSFORMERS                                                       │
  │  ────────────                                                       │
  │   {:publish, _, :hatch}          → cache.put + debounce broadcast   │
  │   {:add_watcher, pid}            → monitor + MapSet.put             │
  │   :flush_broadcast               → local+remote PubSub              │
  │   {:DOWN, _, :process, pid, _}   → remove watcher; empty → stop     │
  │   {:network_change, new_addr}    → Gate.change (addr, new_addr)     │
  │                                                                     │
  │  CLOCK                                                              │
  │  ─────                                                              │
  │   send_after(:flush_broadcast, 100ms)  ← debounce                   │
  │   TTL timer (10 min, external via Cache)                            │
  │                                                                     │
  │  DEATH                                                              │
  │  ─────                                                              │
  │   restart: :temporary                                               │
  │   stop :normal when watchers = ∅                                    │
  │   terminate/2 deletes own cache entry                               │
  └─────────────────────────────────────────────────────────────────────┘
#+END_EXAMPLE

Lens read — /Architect/: the Brahmasthan of this cell is =last=. Everything
else (watchers, timers, reg_key) orbits that emptiness: it is just a map
from event atoms to whatever the Turtle produced. The cell imposes /no
schema/ on its contents — pure Akasha. When a new event type appears, no
code changes here.

Lens read — /Systems/: two balancing loops. (1) Watcher-count loop: watchers
go to zero → cell dies → memory returns to the pool. (2) Debounce loop: N
publishes in 100ms window → 1 broadcast. Both prevent runaway accumulation.

** =Dojo.Gate= — Presence Diffusion Cell
:PROPERTIES:
:MODULE: Dojo.Gate
:END:

#+transclude: [[file:lib/dojo/gate.ex]]  :lines 1-30 :src elixir

=Phoenix.Tracker= with pool_size=2 shards. Each shard is a cell. They gossip
presence deltas among themselves and across nodes via PubSub.

#+BEGIN_EXAMPLE
  ┌────────────────────────── Dojo.Gate (shard) ────────────────────────┐
  │                                                                     │
  │  MEMBRANE                                                           │
  │   track(pid, topic, disciple)   → add to CRDT                       │
  │   change(pid, topic, u, {k,v})  → update metadata                   │
  │   list(topic)                   → query by topic                    │
  │   get_by_key(topic, key)        → lookup                            │
  │                                                                     │
  │  NUCLEUS                                                            │
  │   %{ pubsub_server, node_name } + Tracker's internal CRDT           │
  │                                                                     │
  │  CALLBACK :: handle_diff(diff, state)                               │
  │   For each {topic, {joins, leaves}} ∈ diff:                         │
  │     emit {:join, topic, meta}  via direct_broadcast!                │
  │     emit {:leave, topic, meta} via direct_broadcast!                │
  │                                                                     │
  │  EMERGENT CONTRACT                                                  │
  │   Presence metadata IS the cross-node metadata transport.           │
  │   New observable? Add a meta field. No new wire protocol.           │
  └─────────────────────────────────────────────────────────────────────┘
#+END_EXAMPLE

Lens read — /Educator/: the mental model is accurate all the way down.
"Presence = who is here + what they are doing." Every addition (a cursor,
a volume, a status) is just a key in =meta=. The learner does not have to
relearn anything to add observability.

Lens read — /Adversarial/: failure mode — =broadcast_period= too low under
churn amplifies diffs into storms. Mitigation: pool_size splits work;
Plumtree overload circuit-breaker in =PartisanPubSub.broadcast= drops
remote fan-out while preserving local delivery.

** =Dojo.Cluster.MDNS= — Sensory Cell
:PROPERTIES:
:MODULE: Dojo.Cluster.MDNS
:END:

#+transclude: [[file:lib/dojo/cluster/mdns.ex]]  :lines 1-30 :src elixir

Owns the UDP socket, the peer cache, and the poll heartbeat. This cell
/senses/ the network and /speaks/ on it.

#+BEGIN_EXAMPLE
  ┌───────────────────────── Dojo.Cluster.MDNS ─────────────────────────┐
  │                                                                     │
  │  MEMBRANE                                                           │
  │   cached_peers()   evict_peer(name)   goodbye([ips])                │
  │   reannounce([ips])   diag()                                        │
  │                                                                     │
  │  NUCLEUS                                                            │
  │   %{ socket, adapter, own_name, own_port,                           │
  │      cache :: %{name ⇒ peer_record},                                │
  │      announce_interval  (exponential backoff, cap 15s),             │
  │      poll_cycles, monitored_nodes, socket_retries }                 │
  │                                                                     │
  │  TRANSFORMERS                                                       │
  │   {:udp, socket, ip, port, data}  → parse DNS packet, update cache  │
  │   :poll                           → sweep → announce? → query       │
  │   {:evict_peer, name}             → cache.del + adapter departed    │
  │   {:rejoin_multicast, old, new}   → drop old membership, join new   │
  │   {:nodedown, node}               → reactive failover (accelerator) │
  │                                                                     │
  │  CLOCK                                                              │
  │   poll_interval = 5s                                                │
  │   announce_interval grows 1s → 2s → 4s → 8s → 15s cap               │
  │   POOF : evict if unseen for 4 × poll_interval                      │
  └─────────────────────────────────────────────────────────────────────┘
#+END_EXAMPLE

Lens read — /Systems/: this cell is the canonical /sense-respond loop/.
Stocks: =cache= (peer presence). Flows: announcements outbound, responses
inbound. Delay: POOF tolerance (20s). Balancing: exponential backoff on
announce to prevent multicast flooding.

Watches its own body (the machine's interfaces). The only cell that knows

  ┌──────────────────── Dojo.Cluster.NetworkMonitor ────────────────────┐
  │                                                                     │
  │  NUCLEUS                                                            │
  │   %{ ips, port, pending_ips, stable_count }                         │
  │                                                                     │
  │  DEBOUNCE STATE MACHINE                                             │
  │      same as last          → count ← 0                              │
  │      new, first seen       → pending ← new, count ← 1               │
  │      new, matches pending  → count ← count + 1                      │
  │                               count = 2 → FIRE handle_ip_change     │
  │                                                                     │
  │  THE 7-STEP HOT-SWAP (fired once, atomically-ish)                   │
  │    1. MDNS.goodbye(old_ips)                                         │
  │    2. partisan_config:set(listen_addrs, new)                        │
  │    3. HyParView refresh_node_spec                                   │
  │    4. partisan_sup restart acceptor pool                            │
  │    5. disconnect_stale_peers (via leave, not disconnect)            │
  │    6. rejoin multicast on new ifaces                                │
  │    7. reannounce + cache-addr + broadcast :network_change           │
  └─────────────────────────────────────────────────────────────────────┘
#+END_EXAMPLE

Lens read — /Adversarial/ pre-mortem: what if step 4 (rebind) fails?
→ revert listen_addrs to old, log warning, retry next poll. The cell never
leaves Partisan advertising addresses with no listener bound. /Escape route
exists/.

** Cell Communication — A Signal's Life

One published hatch event, traced from the user pressing Enter in ShellLive
to every remote screen updating. This is the full lifeline of a signal.

#+BEGIN_SRC mermaid :file diagrams/cells-signal-life.png
sequenceDiagram
  autonumber
  participant User as User<br/>(browser)
  participant LV as ShellLive<br/>(LV process)
  participant Turtle as Turtle logic<br/>(in-process)
  participant Table as Table<br/>(per-disciple cell)
  participant Cache as Cache<br/>(Nebulex shard)
  participant PubSub as PubSub<br/>(Partisan adapter)
  participant Plumtree as Plumtree<br/>(gossip)
  participant Remote as Remote Node<br/>(peer cell)

  User->>LV: type command + Enter
  LV->>Turtle: interpret → commands list
  Turtle->>LV: new turtle state (store)
  LV->>Table: cast {:publish, store, :hatch}
  Table->>Cache: async Task.Supervisor<br/>Cache.put(reg_key, :hatch, store)
  Table->>Table: send_after(:flush_broadcast, 100ms)
  Note over Table: debounce window —<br/>further publishes update pending
  Table->>Table: :flush_broadcast tick
  Table->>PubSub: local_broadcast (full payload)<br/>{Dojo.PubSub, :hatch, {reg_key, {Turtle, meta}}}
  PubSub-->>LV: instant render<br/>(same node)
  Table->>PubSub: PubSub.publish (version signal)<br/>%{reg_key, time, state}
  PubSub->>Plumtree: broadcast via Partisan
  Plumtree->>Remote: eager push<br/>(or lazy + graft)
  Remote->>Remote: merge/2 → true (fresh)
  Remote->>Remote: local deliver to subscribers
  Note over Remote: ShellLive on remote node<br/>uses version to request<br/>full payload via RPC
  Remote->>Table: :partisan_rpc.call<br/>local_fetch(topic, :last, :hatch)
  Table-->>Remote: full store
  Remote-->>User: rendered update

  Note over User,Remote: Three layers compose:<br/>• local broadcast (immediate)<br/>• gossip version (cheap fan-out)<br/>• pull-on-demand (authoritative payload)
#+END_SRC

#+RESULTS:
[[file:diagrams/cells-signal-life.png]]

Three layers of functionality emerge from this composition:

1. =Instant local echo=: local subscribers render before the network even
   hears about it. Latency = microseconds.
2. =Version broadcast=: remote nodes learn a new version exists without
   carrying the payload. Latency = gossip hops. Bandwidth = O(peers).
3. =Pull-on-demand=: remote renderers fetch the full state when they
   actually need it. Latency = one RPC. Bandwidth = O(actual viewers).

None of these layers exists in code as a named abstraction. They are
/emergent/ from the composition of cells.

Lens read — /Pragmatist/: "what ships" is layer 1 alone. Dojo could have
stopped there (single-node LOGO) and been useful. Layers 2+3 are what
turned it into a cluster. Each layer was added without breaking the previous.

Lens read — /Systems/: this is the resolution of the polling-vs-reactive
polarity applied to /messages/: cheap gossip is the reactive accelerator,
windowed pull is the safety net, local delivery is immediate. Three
timescales, three bandwidth profiles, all composed from the same primitive
(=send/2=).

Each layer adds exactly one capability over the one below, and only depends
on the /interface/ of what lies beneath — not its identity. You could
replace L3 (Partisan) with classic Erlang distribution and everything above
L3 still works. You could replace L4 (mDNS) with a DNS SRV strategy and
everything above L4 still works. This is what the Architect lens calls
/levels of scale/ and what the Pragmatist calls /reversibility/.
